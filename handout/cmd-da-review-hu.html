Összefoglaló

Egymenetes ördög ügyvédje (devil's advocate) jellegű felülvizsgálat, amely megtalálja az adott célpont — kód, terv, diff vagy design — minden hibáját, anélkül, hogy bármilyen javítást alkalmazna.

Mikor használd

Akkor használd a /da-review parancsot, ha valamiről kritikus, ellenpárti olvasatot szeretnél, és egy prioritizált problémalistát, amellyel utána magad dolgozol tovább. Egyszer fut le, és megáll — nem ciklusol, nem indít javító ügynököket, és nem módosít semmilyen fájlt.

Lényeges különbség az /iterative-review-hoz képest: a /da-review azonosítja a problémákat; az /iterative-review ezzel szemben azonosítja és ki is javítja őket ciklusban mindaddig, amíg nem marad kritikus, súlyos vagy közepes probléma. Akkor válaszd a /da-review-t, ha kézben akarod tartani a javításokat; akkor válaszd az /iterative-review-t, ha automatikusan szeretnéd elhárítani a problémákat.

Tipikus kiváltó helyzetek:

  • Épp befejeztél egy feladat implementálását, és gyors ellenpárti józanságellenőrzést akarsz commit előtt.
  • Egy tervet vagy specifikációt akarsz átvizsgálni hiányosságok után kutatva, mielőtt belevágsz a megvalósításba.
  • Súlyosság szerint csoportosított megállapításokat akarsz anélkül, hogy bármi a kódban változna.
  • Gyors, egymenetes felülvizsgálatra van szükséged, amelyet egy kézi javítási menet bemenetéül használsz.

Meghívás

Add át a felülvizsgálandó célpontot argumentumként. Ha nincs argumentum, akkor a kontextusban éppen lévő tervet vagy kódot vizsgálja át.

/da-review [target]

Paraméterek:

  • [target] — (opcionális) mit vizsgáljon: fájl elérési útja, egy git diff részlet, egy tervfájl elérési útja vagy egy leírás. Ha üres, a parancs az aktuális kontextusban lévő tervet/kódot/munkát vizsgálja.

Engedélyezett eszközök (csak olvasás): Read, Grep, Glob. Az ügynök nem írhat fájlokat, és nem futtathat shell parancsokat.

Súlyossági rubrika (megegyezik az /iterative-review-ban használttal — szinkronban tartva):

  • Kritikus — helyesség, biztonság vagy üzembiztonság szempontjából blokkoló
  • Súlyos — jelentős tervezési hiba, hiányzó követelmény vagy valószínű bug
  • Közepes — nem optimális, de működőképes
  • Apró — stílus, elnevezés vagy kötözködés

1. példa — git diff felülvizsgálata commit előtt (szemléltető példa)

Bemenet:

/da-review $(git diff HEAD)

Kimenet / eredmény:

## Critical
- C1: `process_payment()` calls `charge_card()` without checking the return value.
  If charge fails silently, the order is marked paid without actual payment.

## Major
- M1: No transaction rollback if `save_order()` throws after `charge_card()` succeeds.
  A crash here leaves the customer charged but orderless.

## Moderate
- Mo1: `MAX_RETRY` is hardcoded to 3; should be configurable via env var.

## Minor
- N1: Variable `tmp` in line 47 — rename to `pending_charge` for clarity.

Overall: Critical issue C1 must be fixed before this diff is safe to commit.

2. példa — tervfájl felülvizsgálata hiányosságok után kutatva (szemléltető példa)

Bemenet:

/da-review plans/FEAT-042-auth-redesign.md

Kimenet / eredmény:

## Critical
(none)

## Major
- M1: Task 3.2 adds a new `/logout` endpoint but no task covers session token invalidation on the server side.
  Users who log out remain authenticated until the token expires.
- M2: No rollback plan defined. If the migration in Task 4.1 fails in production, the
  previous auth tables are gone.

## Moderate
- Mo1: Acceptance criteria for Task 2.1 do not mention rate-limiting the login endpoint.

## Minor
- N1: Task numbering jumps from 3.4 to 3.6 — likely a copy-paste error.

Overall: Two major gaps that could cause a security regression — recommend addressing before implementation begins.

Kapcsolódó eszközök

  • /aaa — A minőséget világszínvonalú mércékhez méri kutatás és hiányosságelemzés alapján; akkor használd, ha kiválósági mérőszámot akarsz, nem pusztán hibalistát.
