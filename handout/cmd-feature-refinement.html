/feature-refinement

← Back to index

Summary

An interactive senior product thinker that guides you from a rough feature idea to a well-scoped Feature Brief saved to the project, ready for /plan-maker.

When to use

Use /feature-refinement when you have a feature idea but not yet a clear scope, user flow, or decision log. The command investigates your codebase, challenges your assumptions, and asks targeted questions until the feature is well-defined — then produces a structured Feature Brief file.

It is also described in the agentic workflow pipeline (en.html) as the upstream step before planning and implementation.

Typical trigger situations:

  • You have a rough feature idea and need to scope it before writing a plan.
  • You want an adversarial product thinker to challenge your assumptions before you commit to a direction.
  • You need a Feature Brief file (Documentation/Backlog/…-brief.md) as input for /plan-maker.
  • You want to surface conflicts, risks, or prior decisions from the codebase before designing a solution.

Invocation

Pass your feature idea as the argument. The command starts by investigating the codebase, then enters an iterative clarification loop.

/feature-refinement [feature idea or description]

Parameters:

  • [feature idea or description] — (required) A short description of the feature you want to refine. Can be a sentence, a rough user story, or a few bullet points.

Output: A Feature Brief Markdown file saved to Documentation/Backlog/[feature-name-kebab-case]-brief.md (or project root if that directory does not exist). The brief covers: Problem, Goal, Users & Context, Core Flow, In Scope, Out of Scope, Key Decisions, Edge Cases & Constraints, Open Questions, Future Iterations, and a Recommendation.

Process phases:

  1. Phase 1 — Investigate: Spawns an Explore agent to read all related code, docs, and ADRs.
  2. Phase 2 — Clarify, Challenge & Ideate: Iterative rounds — each round challenges assumptions and asks ≤3 framed questions until scope, flow, and edge cases are resolved.
  3. Phase 3 — Feature Brief: Produces and saves the structured brief, then suggests running /plan-maker [path].

Example 1 — Refine a new notification feature (illustrative example)

Input:

/feature-refinement Add email notifications when a task is assigned to a user

Output / Result:

Investigating the codebase before we start...

Found: existing User model, Task model with assignee_id FK, no email infrastructure yet,
a TODO comment in task_service.py referencing future notifications.

**[Concern: Scope]**
- Option A: Email only — simple, ships fast. Con: will need rework when push notifications arrive.
- Option B: Generic notification system with email as first channel — more upfront work, extensible.
- Recommendation: Option A for now; design the interface to be swappable. Agree?

Q1: Should the assignee receive an email immediately on assignment, or in a daily digest?
  A) Immediate — users get context faster. Con: noisy for high-volume projects.
  B) Configurable per-user — best UX. Con: adds a settings screen to scope.
  C) Immediate for now, configurable later — pragmatic. Recommendation: C.

[After user responds...]

Saved: Documentation/Backlog/task-assignment-notifications-brief.md
Run `/plan-maker Documentation/Backlog/task-assignment-notifications-brief.md` to create an implementation plan.

Example 2 — Scope a search feature with conflicting constraints (illustrative example)

Input:

/feature-refinement Full-text search across all project documents

Output / Result:

Investigating the codebase before we start...

Found: PostgreSQL 15, 3 document types (notes, tasks, files), existing ILIKE queries in search_service.py,
no full-text index, ~50k rows in production. Prior ADR-007 deferred search to "Phase 2".

**[Concern: Risk]**
- ADR-007 deferred this — do you know why? Options:
  A) Performance concern — full-text on 50k rows is fine for pg_trgm; address it.
  B) Scope concern — previous team wanted to ship faster; now it's time.
  Recommendation: Clarify before scoping to avoid repeating the same decision.

Q1: Which document types must be searched in v1?
  A) All three (notes, tasks, files) — complete but more complex.
  B) Notes and tasks only, files deferred — ships faster, covers 90% of use cases.
  Recommendation: B.

[After 2 rounds...]

Saved: Documentation/Backlog/full-text-search-brief.md
Run `/plan-maker Documentation/Backlog/full-text-search-brief.md` to create an implementation plan.

Related tools
