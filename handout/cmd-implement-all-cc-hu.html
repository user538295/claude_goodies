/implement-all-cc (Claude Code változat)

← Vissza az áttekintéshez

Összefoglaló

Az /implement-all Claude Code változata, amely ismételten futtatja az /implement-next-cc parancsot egy tervfájlon, amíg minden feladat el nem készül. A SubagentStop hook addig gátolja a turn lezárását, amíg egy commit le nem landol, és a szülő szintű Case A/B/C/D recovery elcsípi azokat a hibákat, amelyek megkerülik a hookot (OOM, transport hibák, kvótafogyás).

Két változat létezik. Ez az oldal a Claude Code változatot írja le. Egy hook-függetlenül, runtime-függetlenül működő portable változatért (Cursor, claude -p stb.) lásd a /implement-all parancsot.

Mikor használd

Akkor használd az /implement-all-cc parancsot, ha Claude Code-ban futsz, és a legmagasabb megbízhatóságú kezetlen ciklust akarod egy tervfájlon. A hook gate plusz a szülő-szintű recovery elkapja mind a domináns "subagent mid-flow stall" hibatípust, mind a ritkább OOM / kvóta / transport eseteket.

  • Van egy tervfájlod (pl. plans/FEAT-001.md) egy vagy több [ ] feladattal, és sorban mindet implementálni szeretnéd Claude Code-ban.
  • Azt akarod, hogy minden feladat saját, nem üres fájlváltozásokat tartalmazó commitot kapjon (a SubagentStop hook kényszeríti ki).
  • Egy részben befejezett futást folytatsz.
  • Korábban találkoztál a "/implement-next stalled mid-flow" hibatípussal, és auto-rescue-re van szükséged.

Kritikus megszorítások:

  • Megköveteli a SubagentStop hook regisztrációját a ~/.claude/settings.json fájlban (a ~/.claude/scripts/implement-next-stop-gate.sh hívásával).
  • Megköveteli az implement-next-state-write.sh és implement-next-state-clear.sh sentinel scriptek jelenlétét.
  • NEM indít egyetlen ügynököt az összes feladatra — minden feladat külön-külön megy át az /implement-next-cc parancson.

Hívás

Egyetlen argumentumként add át a tervfájl elérési útját.

/implement-all-cc [plan-file-path]

Paraméterek:

  • [plan-file-path] — (kötelező) A terv Markdown-fájljának elérési útja, pl. plans/FEAT-001-my-feature.md. Ha üres vagy nem létezik, a parancs kulcsszó-egyezést keres és megáll egyértelműség nélküli esetekben.

Folyamat (0. lépés + 7 ciklus-lépés + 1 audit):

  1. 0. lépés — Tervfájl feloldása: A Glob tool-lal kulcsszó-egyezett tervfájlt keres, amely legalább egy be nem pipált taskot tartalmaz. Megáll egyértelműség nélkül vagy találat hiányában.

Ciklustörzs (iterációs felső korlát: 50; megakadt-feladat őr: ugyanaz a NEXT_TASK_NAME két iterációban → megáll):

  1. Haladás + megakadt-feladat őr: Lefuttatja a plan-progress.sh-t. Exit 1 → minden feladat kész. Exit 2/3 → hibával megáll. Exit 0 → folytatódik.
  2. Pre-state mentése: rögzíti a START_SHA, START_DIRTY, CWD értékeket.
  3. Háttér Agent indítása: szigorúan szkópolt prompttal, amely így kezdődik: "Call the Skill tool with skill implement-next-cc and args <plan-path>." A prompt PONTOSAN EGY taskot ír elő, megköveteli a 4–8. lépések teljesítését (tesztek zöldek — vagy doc-only task esetén inline verification — plan kipipálva, egyetlen commit, 7. lépés self-verification PASS, 8. lépés report), és tiltja a --no-verify / --amend használatát, task-ok bundle-ozását, beágyazott /implement-all-cc//implement-all//implement-next-cc//implement-next hívásokat és scope creepet. Hiányzó kritérium esetén a subagent TASK INCOMPLETE üzenettel áll meg commit nélkül — a SubagentStop hook blokkolja a turn-endet, a szülő recovery check pedig megállítja a loopot a látható diagnosztikával. A Case B rescue spawn (/implement-next-cc-resume) ekvivalens szigorúan szkópolt prompttal fut, checkoff + commit-re szűkítve (új implementációs munka nélkül). A teljes promptok: commands/implement-all-cc.md Step 3. Rögzíti a szinkron agentId-t.
  4. A SubagentStop gate ÉS a recovery breadcrumb (v2 séma) felfegyverzése: AZUTÁN, hogy az Agent visszaadta az agentId-t, és MIELŐTT a wait-be lépne, a szülő kiírja a teljes v2 breadcrumbot a <CWD>/.claude/implement-next-state.json-ba az implement-next-state-write.sh-n keresztül default módban, valódi agentId-vel és a 6. argumentummal "cc". A breadcrumbnak két fogyasztója van: (a) a SubagentStop hook olvassa és nem engedi lezárni az indított subagent turnjét, amíg új commit nem létezik a START_SHA óta (szűr az agentId-re, hogy a beágyazott sub-sub-agentek átmenjenek); (b) a gyermek 0. lépés triage klasszifikátor minden újraindításkor beolvassa, hogy észlelje a korábbi megszakítást, és az illeszkedő R-esetre dispatch-eljen (R-Fresh / R-A / R-B / R-AB / R-C). A recovery dispatch mátrixért lásd: /implement-next-cc.
  5. Várakozás az agent visszatérésére.
  6. Recovery check — Case A/B/C/D dispatch: FIGYELEM: a lépés elején lévő korábbi feltétel nélküli implement-next-state-clear.sh hívás megszűnt — a gyermek 7. lépés self-verification immár sikeres ágon önmagában törli a breadcrumbot. A szülő csak halt-utakon törli a breadcrumbot (defense-in-depth). Rögzíti az END_SHA/END_DIRTY értékeket, majd dispatch-el:
    • Case A — END_SHA != START_SHA & tiszta munkamásolat → a feladat commitolva, és (a gyermek 7. lépése által) a breadcrumb már törölve; újra plan-progress.sh, folytatódik.
    • Case B — END_SHA == START_SHA & commitolatlan változások → a subagent dolgozott, de a hook valamiért megkerülve (OOM, transport, kvóta). Szülő szinten: (1) teljes tesztsuite Bash + Monitor használatával; (2) tesztbukásra fix agent (≤ 2 retry); (3) ha a tesztek mennek, rescue subagent indul a dedikált /implement-next-cc-resume skill-lel (csak kipipálás + commit a már stage-elt munkán) friss sentinellel; (4) Case A vs B újraértékelése. Ha a rescue után is Case B, megáll — a halt előtt törli a breadcrumbot az implement-next-state-clear.sh-val.
    • Case C — END_SHA == START_SHA & tiszta munkamásolat → a subagent semmit sem csinált. Törli a breadcrumbot, majd diagnosztikával megáll.
    • Case D — END_SHA != START_SHA & commitolatlan változások → anomália (gyakran pre-commit hook hagyott módosított fájlokat). Törli a breadcrumbot, majd megáll.
    • Megakadt-feladat őr halt és iterációs felső korlát halt szintén törlik a breadcrumbot, mielőtt megállnak.
  7. Vissza az 1. lépéshez.

Záró audit (a ciklus terminálása után): lefuttatja az audit-plan-run.sh-t a tervre és a start SHA-ra. Az audit felismeri az új recovery(R-B): és recovery(R-AB): commit-subject prefixeket — ha ezek a markerek megmagyarázzák a commit-szám eltérést, a VIOLATION WARNING-ra fokozódik le (csak szám-eltérés elfogadható recovery commitokra). Az audit kiírja a .claude/recovery-anomalies.log létezését és sorszámát, ha létezik. Az audit befejezése után a szülő törli a breadcrumbot defense-in-depth gyanánt, tiszta futásokra is.

Recovery-check diagnosztika: Cases B, C, D megállásakor a jelentés tartalmazza a START_SHA/END_SHA-t, git status --short-ot, a subagent transcript utolsó 40 sorát (a ~/.claude/projects/.../subagents/agent-<agentId>.jsonl útvonalon), a felismert esetet, és hogy a SubagentStop hook hányszor blokkolt.

Architektúrális megjegyzések: ez a design implementálja az Anthropic által hivatalosan ajánlott "agent stops mid-flow" mintát — egy determinista gate (Stop hook), amely a turn lezárását blokkolja, amíg egy ellenőrző script át nem megy. Layer 1 (SubagentStop hook) az elsődleges védelem; Layer 2 (szülő-szintű recovery + Case B rescue) a biztonsági háló a hookot megkerülő hibákra. A code.claude.com/docs/en/hooks szerint a SubagentStop-hoz nincs dokumentált consecutive-block cap — a 6. lépés Case B/C/D ágai a menekülő útvonal.

Kapcsolódó eszközök

  • /implement-all — Portable runtime-független változat hook-függőség nélkül; használd Cursor-ban, claude -p-ben vagy bármely nem-Claude-Code harness-ben.
  • /implement-next-cc — Az egyfeladatos parancs, amelyre ez a ciklus delegál.
  • /implement-next-cc-resume — A dedikált rescue skill, amelyet ennek a ciklusnak a Case B útja hív; csak kipipálást + committot végez a már stage-elt munkán.
  • /implement-next — Az /implement-next-cc portable párja.
  • /iterative-review — A DA review-ciklus, amelyet minden /implement-next-cc task implementáció belül használ.
  • scripts-plan — Tartalmazza a plan-progress.sh, audit-plan-run.sh, implement-next-stop-gate.sh, implement-next-state-write.sh, implement-next-state-clear.sh szkripteket.
