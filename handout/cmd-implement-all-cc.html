/implement-all-cc (Claude Code variant)

← Back to index

Summary

Claude Code variant of /implement-all that repeatedly runs /implement-next-cc on a plan file until every task is complete, with a SubagentStop hook blocking turn-end until a commit lands and parent-level Case A/B/C/D recovery for failures that bypass the hook (OOM, transport errors, quota hits).

Two variants exist. This page describes the Claude Code variant. For a portable runtime-agnostic version without hook dependency (works in Cursor, claude -p, etc.), see /implement-all.

When to use

Use /implement-all-cc when you are running inside Claude Code and want the highest-reliability hands-off loop over a plan file. The hook gate plus parent-level recovery catches both the dominant "subagent stalled mid-flow" failure mode and the rarer OOM / quota / transport edge cases.

  • You have a plan file (e.g. plans/FEAT-001.md) with one or more [ ] tasks and want to implement them all in sequence in Claude Code.
  • You want each task to get its own commit with non-empty file changes (enforced by the SubagentStop hook).
  • You are resuming a partially-completed run and want to continue from the next uncompleted task.
  • You have hit the "implement-next stalled mid-flow" failure mode before and need auto-rescue.

Critical constraints:

  • Requires the SubagentStop hook to be registered in ~/.claude/settings.json (calling ~/.claude/scripts/implement-next-stop-gate.sh).
  • Requires the sentinel scripts implement-next-state-write.sh and implement-next-state-clear.sh to be present.
  • Does NOT spawn a single agent for all tasks — every task goes through /implement-next-cc individually.

Invocation

Pass the path to the plan file as the sole argument.

/implement-all-cc [plan-file-path]

Parameters:

  • [plan-file-path] — (required) Path to the plan Markdown file, e.g. plans/FEAT-001-my-feature.md. If blank or non-existent, the command searches for a keyword-matched candidate and halts on ambiguity.

Process (Step 0 + 7 loop steps + 1 audit):

  1. Step 0 — Resolve the plan file: Uses the Glob tool to find a keyword-matched plan file with at least one unchecked task. Halts on ambiguity or no match.

Loop body (iteration cap: 50; stuck-task guard: same NEXT_TASK_NAME two iterations in a row halts):

  1. Progress + stuck-task guard: Runs plan-progress.sh. Exit 1 → all tasks complete, stop. Exit 2/3 → halt with error. Exit 0 → continue.
  2. Capture pre-state: records START_SHA, START_DIRTY, CWD.
  3. Spawn background Agent: with the prompt "Call the Skill tool with skill implement-next-cc and args <plan-path>." Records the synchronous agentId.
  4. Arm the SubagentStop gate AND the recovery breadcrumb (v2 schema): AFTER Agent returns the agentId and BEFORE the wait, the parent writes the full v2 breadcrumb to <CWD>/.claude/implement-next-state.json via implement-next-state-write.sh in default mode with the real agentId and the 6th arg "cc". The breadcrumb serves two consumers: (a) the SubagentStop hook reads it and refuses to let the spawned subagent end its turn until a new commit exists since START_SHA (filters on agentId so nested sub-sub-agents pass through); (b) the child's Step 0 triage classifier reads it on every re-invocation to detect a prior interruption and dispatch to the matching R-case (R-Fresh / R-A / R-B / R-AB / R-C). See /implement-next-cc for the recovery dispatch matrix.
  5. Wait for the agent to return.
  6. Recovery check — Case A/B/C/D dispatch: NOTE: the prior unconditional implement-next-state-clear.sh at the start of this step has been removed — the child's Step 7 self-verification now clears the breadcrumb on its own success path. The parent only clears the breadcrumb on halt paths (defense-in-depth). Captures END_SHA/END_DIRTY and dispatches:
    • Case A — END_SHA != START_SHA & tree clean → task committed and (per child Step 7) breadcrumb already cleared; re-run plan-progress.sh and continue (or halt if task not checked off).
    • Case B — END_SHA == START_SHA & uncommitted changes → subagent did work but hook bypassed (OOM, transport, quota). At parent level: (1) run the full test suite using Bash + Monitor; (2) if tests fail, spawn fix agent (≤ 2 retries); (3) if tests pass, spawn rescue subagent that invokes the dedicated /implement-next-cc-resume skill (performs only check-off + commit on the already-staged work) with a fresh sentinel; (4) re-evaluate Case A vs B once more. If still Case B after the rescue, halt — clear the breadcrumb via implement-next-state-clear.sh before the halt.
    • Case C — END_SHA == START_SHA & tree clean → subagent did nothing visible. Clear the breadcrumb, then halt with diagnostic.
    • Case D — END_SHA != START_SHA & uncommitted changes → anomaly (often pre-commit hook left files modified). Clear the breadcrumb, then halt.
    • Stuck-task guard halt and iteration-count-exceeded halt also clear the breadcrumb before halting.
  7. Return to step 1.

Final audit (after the loop terminates): runs audit-plan-run.sh against the plan and the start SHA. The audit recognizes the new recovery(R-B): and recovery(R-AB): commit-subject prefixes — when these markers explain the commit-count delta, the VIOLATION is downgraded to WARNING (count-only mismatch is acceptable for recovery commits). The audit also surfaces the existence and line count of .claude/recovery-anomalies.log if present. After the audit completes, the parent clears the breadcrumb as defense-in-depth for clean runs.

Recovery-check diagnostic: when Cases B, C, or D halt, the report includes START_SHA/END_SHA, git status --short, last 40 lines of the subagent transcript (located via the ~/.claude/projects/.../subagents/agent-<agentId>.jsonl path), the detected case, and how many times the SubagentStop hook blocked.

Architectural notes: this design implements Anthropic's officially-prescribed "agent stops mid-flow" pattern — a deterministic gate (Stop hook) that blocks turn-end until a verification script passes. Layer 1 (SubagentStop hook) is the primary defense; Layer 2 (parent-level recovery + Case B rescue) is the safety net for failures that bypass the hook. Per code.claude.com/docs/en/hooks, no consecutive-block cap is documented for SubagentStop — Case B/C/D in Step 6 is the user's escape hatch.

Example 1 — Run all tasks with hook-gated execution (illustrative example)

Input:

/implement-all-cc plans/FEAT-001-skill-command-script-handouts.md

Output / Result:

[Iteration 1] plan-progress.sh → 0/7 complete, next: Task 3.3 — cmd-implement-all.html
  → spawning /implement-next-cc background agent (agentId: a1b2c3)…
  → SubagentStop hook armed with sentinel at .claude/implement-next-state.json
  → agent returned; END_SHA differs; tree clean → Case A
  → sentinel cleared; commit b1d2e3f recorded

[Iteration 2] plan-progress.sh → 1/7 complete, next: Task 3.4 — cmd-implement-next.html
  → spawning /implement-next-cc background agent (agentId: d4e5f6)…
  → Stop hook fired once (blocked premature turn-end), then commit landed
  → Case A; commit c2e3f40

… (5 more iterations) …

[Iteration 8] plan-progress.sh → All tasks complete. (exit 1)
All tasks complete.

Example 2 — Case B rescue after stalled subagent (illustrative example)

Input:

/implement-all-cc plans/FEAT-002-api-refactor.md

Output / Result:

[Iteration 3] plan-progress.sh → 5/7 complete, next: Task 2.4 — Wire new AuthService
  → spawning /implement-next-cc background agent (agentId: 9z8y7x)…
  → agent returned; END_SHA == START_SHA but tree dirty (3 files staged) → Case B
  → sentinel cleared; running full test suite at parent level…
    pytest: 142 passed, 0 failed
  → spawning rescue subagent: /implement-next-cc-resume <plan>
  → fresh sentinel armed (agentId: rescue-1)
  → rescue agent committed; re-evaluated → Case A
  → commit e5f6071 recorded

[Iteration 4] plan-progress.sh → 6/7 complete, next: Task 2.5 — Update integration tests
  → … (continues normally)

Related tools

  • /implement-all — Portable runtime-agnostic version without hook dependency; use this in Cursor, claude -p, or any non-Claude-Code harness.
  • /implement-next-cc — The single-task command that this loop delegates to.
  • /implement-next-cc-resume — The dedicated rescue skill that this loop's Case B path invokes; performs only check-off + commit on already-staged work.
  • /implement-next — Portable sibling of /implement-next-cc.
  • /iterative-review — The DA review loop invoked inside each /implement-next-cc task implementation.
  • scripts-plan — Includes plan-progress.sh, audit-plan-run.sh, implement-next-stop-gate.sh, implement-next-state-write.sh, implement-next-state-clear.sh.
