/implement-all (portable variant)

← Back to index

Summary

Portable runtime-agnostic loop that delegates to /implement-next — one task at a time — until every uncompleted task in a plan file is implemented and committed. Works in any harness (Claude Code, Cursor, claude -p) without hook dependencies.

Two variants exist. This page describes the portable variant. For Claude Code's hook-enforced variant with auto-rescue (Case A/B/C/D recovery + SubagentStop gate), see /implement-all-cc.

Using in Cursor (one-time setup)

Cursor reads custom commands from .cursor/commands/ in your project, not from ~/.claude/commands/. To use this skill in Cursor, copy or symlink the portable command files:

mkdir -p .cursor/commands
ln -s ~/.claude/commands/implement-all.md .cursor/commands/implement-all.md
ln -s ~/.claude/commands/implement-next.md .cursor/commands/implement-next.md

Cursor's Task tool primitive translates equivalently to Claude Code's Agent. The portable variant does NOT use hooks — no further setup.

When to use

Use /implement-all when you have a plan file with multiple uncompleted tasks and want to run the full implementation loop hands-off. The command loops until the plan is complete — spawning one agent per task, each delegating to /implement-next — without requiring manual intervention between tasks.

  • You have a plan file (e.g. plans/FEAT-001.md) with one or more [ ] tasks and want to implement them all in sequence.
  • You want each task to get its own commit with non-empty file changes (enforced by /implement-next).
  • You are resuming a partially-completed run and want to continue from the next uncompleted task.

Critical constraints (enforced by the command itself):

  • Does NOT spawn a single agent for all tasks — every task goes through /implement-next individually.
  • Does NOT write code directly — all implementation is delegated.
  • Does NOT proceed if the plan's task section heading is missing (raises a WARNING and stops).

Invocation

Pass the path to the plan file as the sole argument.

/implement-all [plan-file-path]

Parameters:

  • [plan-file-path] — (required) Relative path to the plan Markdown file, e.g. plans/FEAT-001-my-feature.md.

Process (Step 0 + 5 loop steps):

  1. Step 0 — Resolve the plan file: If the argument is a path to an existing file, use it. Otherwise, glob **/*.md and keyword-match candidates with at least one unchecked task. Halt on ambiguity or no match.

Loop body (iteration cap: 50; stuck-task guard: same NEXT_TASK_NAME two iterations in a row halts):

  1. Progress + stuck-task guard: Runs bash ~/.claude/scripts/plan-progress.sh <plan-file>. Exit 1 → all tasks complete, stop. Exit 2/3 → halt with error. Exit 0 → note NEXT_TASK_NAME and continue.
  2. Capture pre-state: records START_SHA and CWD. On iteration 1 only, writes the start SHA to $CWD/.claude/implement-all-start-sha-<8-hex-hash-of-plan-path> so step 5's audit doesn't depend on a transient variable. Each plan gets its own SHA-anchor file (namespaced by hash of the plan path), so different plans don't collide on halt-and-resume.
  3. Spawn subagent + arm recovery breadcrumb: uses the runtime's subagent primitive (Claude Code Agent tool, Cursor Task tool, or inline /implement-next in headless harnesses) with the prompt: "Run /implement-next on plan file <plan-path>." Immediately after spawning (and BEFORE the wait), writes the recovery breadcrumb via implement-next-state-write.sh --upsert "$CWD" "$START_SHA" <plan-path> "$NEXT_TASK_NAME" "" "portable". The portable variant has no SubagentStop hook to coordinate with, so --upsert mode with empty expected_agent_id is used: the breadcrumb still enables the child's Step 0 triage to detect a prior interruption and dispatch to the matching R-case (R-Fresh / R-A / R-B / R-AB / R-C / R-Halt / R-Stuck). See /implement-next for the recovery dispatch matrix. Then waits for return.
  4. Recovery check — verify the task landed: portable substitute for hook enforcement. Halts (does not auto-rescue) on any of these failure modes — and on every halt path, clears the breadcrumb via implement-next-state-clear.sh "$CWD" before the halt:
    • END_SHA != START_SHA & tree clean → likely committed (child Step 7 already cleared the breadcrumb); re-run plan-progress.sh and continue (or halt + clear breadcrumb if task not checked off).
    • END_SHA == START_SHA & uncommitted changes → subagent stalled mid-flow (the dominant historical failure mode); clear breadcrumb, halt with diagnostic. Suggests /implement-all-cc for Claude Code.
    • END_SHA == START_SHA & tree clean → subagent did nothing visible (network, OOM, quota). Clear breadcrumb, halt.
    • END_SHA != START_SHA & uncommitted changes → anomaly; clear breadcrumb, halt.
    • Stuck-task guard (same NEXT_TASK_NAME twice) and iteration-count exceeded both clear the breadcrumb before halt.
  5. Audit: after the loop terminates, runs bash ~/.claude/scripts/audit-plan-run.sh <plan-path> <FIRST_START_SHA>, then cleans up $CWD/.claude/implement-all-start-sha-<8-hex-hash-of-plan-path> AND clears the breadcrumb (defense-in-depth). The audit recognizes the new recovery(R-B): and recovery(R-AB): commit-subject prefixes — when these markers explain the commit-count delta, the VIOLATION is downgraded to WARNING. The audit also surfaces the existence and line count of .claude/recovery-anomalies.log if present. Exit 0 = clean run; exit 1 = mismatch.

Why this variant is less reliable than -cc: relies on the subagent following the /implement-next markdown instructions to commit before ending its turn. The user's archon-search transcripts measured ~7.6% of /implement-next subagents historically never committing (the dominant failure mode this skill addresses). Plan-and-Solve (arxiv.org/abs/2305.04091, accessed 2026-06-11) qualitatively names "missing-step errors" as a known failure class for prompt-only multi-step flows — our measurement is a concrete instance of that class. Halting bounds the damage but does not auto-recover.

Example 1 — Run all tasks in a feature plan (illustrative example)

Input:

/implement-all plans/FEAT-001-skill-command-script-handouts.md

Output / Result:

[Iteration 1] plan-progress.sh → 0/7 complete, next: Task 3.3 — cmd-implement-all.html
  → spawning /implement-next agent… done (commit b1d2e3f)

[Iteration 2] plan-progress.sh → 1/7 complete, next: Task 3.4 — cmd-implement-next.html
  → spawning /implement-next agent… done (commit c2e3f40)

… (5 more iterations) …

[Iteration 8] plan-progress.sh → All tasks complete. (exit 1)
All tasks complete.

Example 2 — Resume after a partial run (illustrative example)

Input:

/implement-all plans/FEAT-002-api-refactor.md

Output / Result:

(4 tasks already marked [x] from a prior session)

[Iteration 1] plan-progress.sh → 4/7 complete, next: Task 2.3 — Extract AuthService
  → spawning /implement-next agent… done (commit d4e5f60)

[Iteration 2] plan-progress.sh → 5/7 complete, next: Task 2.4 — Wire new AuthService
  → spawning /implement-next agent… done (commit e5f6071)

[Iteration 3] plan-progress.sh → 6/7 complete, next: Task 2.5 — Update integration tests
  → spawning /implement-next agent… done (commit f607182)

[Iteration 4] plan-progress.sh → All tasks complete. (exit 1)
All tasks complete.

Related tools

  • /implement-all-cc — Claude Code variant with SubagentStop hook enforcement and Case A/B/C/D parent-level recovery; use when running in Claude Code for higher reliability.
  • /implement-next — The portable single-task implementation command that /implement-all delegates to for each task; runs TDD, optional critical review, tests, check-off, commit, and self-verification.
  • /implement-next-cc — Claude Code variant of /implement-next; paired with /implement-all-cc and gated by the SubagentStop hook.
  • /iterative-review — Devil's advocate review loop invoked by /implement-next during each task's implementation; spawns parallel DA agents and fix agents until no critical/major/moderate issues remain.
  • scripts-plan — The plan-progress.sh and audit-plan-run.sh scripts called by this command.
