/implement-next-cc-resume (Claude Code rescue skill)

← Back to index

Summary

Dedicated rescue skill for the Case-B path of /implement-all-cc. Performs only check-off + commit on already-staged work — never re-implements, never re-runs tests, never re-reviews. The parent loop spawns this skill after it has independently verified (at parent level) that the implementation in the working tree is complete and the full test suite is green; the rescue agent just needs to mark the task [x] in the plan and commit the staged changes.

Do NOT invoke this skill directly. It is meant only to be called by /implement-all-cc's Case-B rescue path. Direct invocation will (at best) commit whatever the working tree happens to contain — without the parent's pre-checks, the result is unsafe.

When this skill runs

/implement-all-cc spawns this skill exactly when it detects Case B during its recovery dispatch:

  • END_SHA == START_SHA & the working tree has uncommitted changes — the prior /implement-next-cc subagent did implementation work but never landed a commit (OOM, transport error, quota hit, or the SubagentStop hook was bypassed).
  • The parent has already run the full test suite at parent level using Bash + Monitor and confirmed tests pass.
  • The parent has armed a fresh sentinel for this rescue agent's agentId.

Under those conditions, the only thing left to do is (a) update the plan file's checkbox for the most-recently-implemented task and (b) commit. That is exactly — and only — what this skill does.

Invocation

Pass the path to the plan file as the sole argument (no flags).

/implement-next-cc-resume [plan-file-path]

Parameters:

  • [plan-file-path] — (required) Path to the plan Markdown file whose task is being rescued, e.g. plans/FEAT-002-api-refactor.md.

Steps executed (3 steps total):

  1. Step 1 — Identify the task: Runs plan-progress.sh to get NEXT_TASK_NAME. The first unchecked task in the plan is the one to check off. (This is deterministic; the dirty tree may contain code for the task but the file-list alone does NOT uniquely identify which task it is.)
  2. Step 2 — Commit: Stages implementation files AND the updated plan file, then commits with a message derived from the task content. NON-NEGOTIABLE: one task = one commit. Pre-commit hooks must NOT be skipped. If the commit fails, the skill reports failure to the parent — the parent will halt.
  3. Step 3 — Report: Emits a single-line report: Rescue commit landed: <sha> on success, or Rescue failed: <reason> on failure.

Why a separate skill instead of a flag on /implement-next-cc: An earlier design used a resume flag on the main /implement-next-cc skill to switch into rescue mode. The C5 refactor split this out into a dedicated skill so the main implementation skill stays single-purpose and the rescue path is impossible to trigger by accident from a flag-typo.

Example 1 — Rescuing a stalled task (illustrative example)

Input (as invoked by /implement-all-cc's Case B rescue):

Call the Skill tool with skill `implement-next-cc-resume` and args `plans/FEAT-002-api-refactor.md`

Output / Result:

[Rescue skill: parent has verified implementation + tests pass]
[Skipping any implementation / DA review / test work — checkoff + commit only]

[Step 1] Running plan-progress.sh → NEXT_TASK_NAME=Task 2.4 — Wire new AuthService
           Plan updated: Task 2.4 marked [x]

[Step 2] Staging changes + updated plan, committing…

           SubagentStop hook verified new commit since START_SHA → turn-end allowed

[Step 3] Rescue commit landed: e5f6071
