/implement-next-cc (Claude Code variant)

← Back to index

Summary

Claude Code variant of /implement-next: implements one task from a plan file with TDD, parallel DA reviews, blocking tests, check-off, and commit. Paired with /implement-all-cc whose SubagentStop hook blocks turn-end until a commit lands. Case B rescue at the parent level is handled by the dedicated /implement-next-cc-resume skill — never invoke that skill directly.

Two variants exist. This page describes the Claude Code variant. For a portable runtime-agnostic version without hook dependency, see /implement-next.

When to use

Use /implement-next-cc when you are running inside Claude Code and want one task implemented with full quality gates (TDD, DA review, test pass, commit) and protected by the SubagentStop hook that blocks turn-end before a commit. This is the atomic unit invoked by /implement-all-cc.

  • You have a plan file and want to implement only the next task in Claude Code.
  • You want hook-enforced commit-before-turn-end behavior.
  • You are stepping through a plan manually inside Claude Code, one task at a time.
  • You are debugging a failed /implement-all-cc run and want to re-run a single task in isolation.

Relationship to /implement-all-cc: /implement-all-cc spawns one /implement-next-cc background agent per uncompleted task and arms the SubagentStop hook with a sentinel filtered by agentId. The hook blocks the subagent's turn-end until a new commit exists since START_SHA. Failures that bypass the hook (OOM, transport errors, quota hits) are caught by the parent's Case A/B/C/D recovery.

Rescue mode: when /implement-all-cc detects a stalled subagent (Case B), it invokes the dedicated /implement-next-cc-resume skill which performs only check-off + commit on the already-staged work. The rescue skill is a separate command — this skill is no longer flag-driven.

Invocation

Pass the path to the plan file as the sole argument.

/implement-next-cc [plan-file-path]

Parameters:

  • [plan-file-path] — (required) Path to the plan Markdown file, e.g. plans/FEAT-001-my-feature.md.

Note on rescue: For Case B rescue (parent has verified implementation + tests but no commit landed), the parent uses a separate skill, /implement-next-cc-resume, which performs only check-off + commit on the already-staged work. Do not pass any "resume" flag to this command — that flag no longer exists.

Steps executed for each task (7 steps total):

  1. Step 1 — Show progress: Runs plan-progress.sh to display current completion and identify the next uncompleted task.
  2. Step 2 — Implement (TDD): Spawns the most appropriate agent to implement only this task. For code tasks: failing tests first (red), implementation (green), full pass. For non-code tasks: implements directly.
  3. Step 3 — DA review loop: Invokes /iterative-review with git diff HEAD and the task spec. Includes the loud "MANDATORY CONTINUATION — do not stop here" warning to defend against the subagent ending its turn at the review verdict instead of proceeding to Step 4. (The SubagentStop hook is a secondary defense — see Step 4.)
  4. Step 4 — Run tests: Subagent-specific guidance — read carefully:
    • Tests complete in ≤ 9 minutes (preferred): run the full suite with Bash foreground, timeout: 540000. Block until it returns.
    • Tests do NOT fit in 9 minutes: run only the task-relevant subset and end the turn with the working tree uncommitted but the relevant subset green. /implement-all-cc's Case B recovery at the parent level runs the full suite there (where Monitor works correctly).
    Never use the Monitor tool inside this subagent to wait for tests — Monitor tells the agent to yield, but a subagent has no future user turn for events to attach to. The run silently dies and the parent sees an empty commit. This was the dominant failure mode of historical implement-next runs.
  5. Step 5 — Check off: Spawns an agent to mark the implemented task and all completed sub-items [x] in the plan file.
  6. Step 6 — Commit: Stages implementation files and the updated plan file; commits with a message derived from the task content. --no-verify is forbidden. The SubagentStop hook (when invoked from /implement-all-cc) reads the sentinel at $CWD/.claude/implement-next-state.json and refuses to let this subagent end its turn until a new commit exists since START_SHA.
  7. Step 7 — Report: Outputs a concise summary: what was implemented, test results, deviations, what was checked off.

Example 1 — Hook-gated single-task implementation (illustrative example)

Input (as invoked by /implement-all-cc):

Call the Skill tool with skill `implement-next-cc` and args `plans/FEAT-001-skill-command-script-handouts.md`

Output / Result:

Progress   : [████████░░░░] 68%  (13/19 tasks, 2/6 phases)
Next task  : 3.4  **File**: `handout/cmd-implement-next-cc.html`

NEXT_TASK_NAME=**File**: `handout/cmd-implement-next-cc.html`
NEXT_TASK_PHASE=3
NEXT_TASK_NUMBER=4

[Step 2] Spawning implementation agent for Task 3.4…
  → Reads commands/implement-next-cc.md
  → Writes handout/cmd-implement-next-cc.html

[Step 3] Invoking /iterative-review…
  → DA agent: no critical/major/moderate issues found (PASS)

[Step 4] Full test suite (timeout 540000 ms): all structural checks pass (CLEAN)

[Step 5] Plan updated: Task 3.4 marked [x]

[Step 6] Commit: feat(handout): Task 3.4 — cmd-implement-next-cc.html
  → SubagentStop hook verified new commit since START_SHA → turn-end allowed

[Step 7] Report: Task 3.4 complete.

Example 2 — Case B rescue via separate /implement-next-cc-resume skill (illustrative example)

Input (as invoked by /implement-all-cc's Case B rescue — not this skill, but the rescue skill):

Call the Skill tool with skill `implement-next-cc-resume` and args `plans/FEAT-002-api-refactor.md`

Output / Result:

[Rescue skill: parent has verified implementation + tests pass — performing only check-off + commit]

[Step 1] Plan updated: Task 2.4 marked [x]

[Step 2] Commit: feat(auth): Task 2.4 — Wire new AuthService
  → SubagentStop hook verified new commit since START_SHA → turn-end allowed

[Step 3] Report: Rescue commit landed: e5f6071

See /implement-next-cc-resume for the rescue skill's full documentation.

Related tools

  • /implement-next-cc-resume — Dedicated rescue skill called by /implement-all-cc's Case B path; performs only check-off + commit on already-staged work. Replaces the old in-flag rescue mode of this skill.
  • /implement-next — Portable runtime-agnostic sibling without hook dependency; use this outside Claude Code.
  • /implement-all-cc — The Claude Code parent loop that delegates to this command and arms the SubagentStop gate.
  • /implement-all — Portable parent-loop sibling.
  • /iterative-review — The DA review loop invoked in Step 3.
  • scripts-plan — Includes plan-progress.sh (Step 1) plus the hook + sentinel scripts implement-next-stop-gate.sh, implement-next-state-write.sh, implement-next-state-clear.sh.
