/implement-next (portable variant)

← Back to index

Summary

Portable runtime-agnostic command that reads a plan file, implements the next uncompleted task with strict TDD, an optional critical review, runs tests blocking, checks off the completed task, commits, and self-verifies the commit before ending — one task, one commit. Works in any harness without hook dependencies.

Two variants exist. This page describes the portable variant. For Claude Code's hook-enforced variant (paired with /implement-all-cc and gated by the SubagentStop hook), see /implement-next-cc.

Using in Cursor: Cursor reads commands from .cursor/commands/, not ~/.claude/commands/. See the Cursor setup section in cmd-implement-all.html for the one-time symlink instructions.

When to use

Use /implement-next when you want to implement exactly one task from a plan file with full quality gates: TDD cycle, DA review loop, test suite pass, and a clean commit. It is the atomic unit invoked by /implement-all in a loop.

  • You have a plan file with one or more [ ] tasks and want to implement only the next one.
  • You want automated TDD enforcement: tests written first (red), then implementation (green), then full suite pass before committing.
  • You want a DA review loop to catch issues before the commit lands.
  • You are stepping through a plan manually, one task at a time, to inspect output between tasks.
  • You are debugging a failed /implement-all run and want to re-run a single task in isolation.

Relationship to /implement-all: /implement-all delegates to /implement-next for every uncompleted task in the plan. You can invoke /implement-next directly to implement one task at a time, or let /implement-all drive the loop automatically.

Relationship to scripts-plan: /implement-next calls plan-progress.sh in Step 1 to identify the next task and display progress, and calls check-task-commit.sh in Step 7 as part of its self-verification (the portable substitute for the SubagentStop hook used in the -cc variant).

Invocation

Pass the path to the plan file as the sole argument.

/implement-next [plan-file-path]

Parameters:

  • [plan-file-path] — (required) Relative path to the plan Markdown file, e.g. plans/FEAT-001-my-feature.md.

Steps executed for each task (9 steps total — Step 0 triage + 8 work steps):

  1. Step 0 — Triage prior state (recovery dispatch): Shells out to implement-next-triage.sh "$(pwd)" "$ARGUMENTS" "portable". On exit 0, parses CASE=, START_SHA, START_CHECKED, SHA_BEFORE, BRANCH_NAME, TASK_NAME, REVIEW_RANGE, STEP_2_RESUME, REVIEW_ABORT_COUNT. On exit 1 (R-Halt / R-Stuck), prints the diagnostic and exits the turn immediately. Note: START_SHA and START_CHECKED capture has moved from the legacy Step 1 to Step 0 so recovery paths that skip Step 1 still bind these values for Step 7's self-verification. Dispatch:
    • R-Fresh → proceed to Step 1.
    • R-A (uncommitted partial impl) → SKIP Steps 1 & 2; jump to Step 3 with REVIEW_RANGE=HEAD+worktree.
    • R-B (committed but plan not checked off) → SKIP Steps 1 & 2; insert plan-file checkoff for TASK_NAME; review <sha_before>..HEAD; Step 6's commit subject MUST begin with recovery(R-B):.
    • R-AB (hybrid R-A + R-B) → SKIP Steps 1 & 2; review <sha_before>..HEAD+worktree; Step 6's commit subject MUST begin with recovery(R-AB):.
    • R-C (pre-impl TDD-red state) → SKIP Step 1; resume Step 2 on sub-items still unchecked in git show HEAD:$ARGUMENTS.
  2. Step 1 — Show progress: Runs plan-progress.sh to display current completion and identify the next uncompleted task. If all tasks are complete (exit 1), stops and reports completion. (START_SHA / START_CHECKED capture has moved to Step 0.)
  3. Step 2 — Implement (TDD): Spawns a focused subagent (or proceeds inline if the runtime has no subagent primitive) to implement only this task. For code tasks: failing tests first (red), then implementation (green), then full pass. For non-code tasks (docs, config): implements directly.
  4. Step 3 — Critical review (optional): Invokes /iterative-review on the diff (use the REVIEW_RANGE from Step 0 if set, otherwise git diff HEAD) plus the task spec, or spawns one critic subagent if the slash command is unavailable. Includes the loud "MANDATORY CONTINUATION — do not stop here" warning: the portable variant has no SubagentStop hook to catch this, so the prompt is the only defense against the subagent ending its turn at the review verdict. Review-abort handling: if the review aborts mid-flow, restart it ONCE. If the restart also aborts: run implement-next-state-write.sh --increment-review-abort "$(pwd)", then HALT — do NOT commit. Next invocation's Step 0 will see review_abort_count >= 2 and dispatch R-Stuck.
  5. Step 4 — Run tests: Runs the full test suite as a blocking foreground command, timeout under the runtime's foreground ceiling (Claude Code's Bash ≤ 600,000 ms). If the full suite cannot complete in one blocking call, runs only the task-relevant subset and reports the partial scope in Step 8. Fix agents retry up to 3 times.
  6. Step 5 — Check off: Marks the implemented task and all completed sub-items [x] in the plan file.
  7. Step 6 — Commit: Stages implementation files and the updated plan file; commits with a message derived from the task content. --no-verify is forbidden. Includes recovery on commit failure (unstage, restore plan file, fix the underlying issue, redo Steps 5–6). Recovery-case prefixes: if Step 0 dispatched R-B, the commit subject MUST begin with recovery(R-B):; if R-AB, with recovery(R-AB):. audit-plan-run.sh matches these anchored prefixes and downgrades VIOLATION to WARNING.
  8. Step 7 — Self-verification (portable substitute for hook enforcement): Programmatically verifies before ending the turn: (1) check-task-commit.sh "$START_SHA" confirms a new commit exists (for R-B, START_SHA is the breadcrumb's sha_before); (2) plan's checked-task count grew by at least 1; (3) the new commit's diff includes both the plan file AND implementation files — UNLESS the latest commit's subject begins with recovery(R-B): (R-B's commit is plan-only by design; the check self-skips). On success, clears the breadcrumb via implement-next-state-clear.sh "$(pwd)". If any check fails, does NOT end the turn — diagnoses the gap and returns to the appropriate earlier step.
  9. Step 8 — Report: Outputs a concise summary: what was implemented, test results, any deviations, what was checked off, Step 7's self-verification result.

Example 1 — Implement the next task in a feature plan (illustrative example)

Input:

/implement-next plans/FEAT-001-skill-command-script-handouts.md

Output / Result:

Progress   : [████████░░░░] 68%  (13/19 tasks, 2/6 phases)
Next task  : 3.4  **File**: `handout/cmd-implement-next.html`

NEXT_TASK_NAME=**File**: `handout/cmd-implement-next.html`
NEXT_TASK_PHASE=3
NEXT_TASK_NUMBER=4

[Step 2] Spawning implementation agent for Task 3.4…
  → Reads commands/implement-next.md
  → Writes handout/cmd-implement-next.html with 7 sections and 2 examples
  → Pre-check tests pass (section count=7, source comment present, etc.)

[Step 3] Invoking /iterative-review…
  → DA agent: no critical/major/moderate issues found (PASS)

[Step 4] Full test suite: 14 structural checks pass (CLEAN)

[Step 5] Plan updated: Task 3.4 marked [x]

[Step 6] Commit: feat(handout): Task 3.4 — cmd-implement-next.html
  → 1 file changed: handout/cmd-implement-next.html (new)
  → 1 file changed: plans/FEAT-001-skill-command-script-handouts.md (task checked off)

[Step 7] Report: Task 3.4 complete. 7 sections, 2 examples, forward references noted
  (cmd-iterative-review.html, scripts-plan.html not yet created — will resolve by Task 6.1).

Example 2 — All tasks complete, command halts early (illustrative example)

Input:

/implement-next plans/FEAT-002-api-refactor.md

Output / Result:

plan-progress.sh exit code: 1

All tasks in plans/FEAT-002-api-refactor.md are complete.
Nothing to implement.

Summary: 0 tasks implemented. Plan is fully checked off.

Related tools

  • /implement-next-cc — Claude Code variant of this command, paired with /implement-all-cc and gated by the SubagentStop hook; Case B rescue is handled by the separate /implement-next-cc-resume skill.
  • /implement-all — Portable parent loop that invokes /implement-next once per uncompleted task; use this when you want fully automated hands-off execution of all remaining tasks.
  • /implement-all-cc — Claude Code parent loop with hook-enforced commit gate and Case A/B/C/D recovery.
  • /iterative-review — The optional critical-review loop invoked in Step 3; spawns parallel DA agents and fix agents until no critical/major/moderate issues remain.
  • scripts-plan — The plan-progress.sh (Step 1) and check-task-commit.sh (Step 7) scripts used by this command.
