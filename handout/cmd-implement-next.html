/implement-next

← Back to index

Summary

Reads a plan file, implements the next uncompleted task using agents with strict TDD, runs parallel DA reviews and fixes all issues, checks off the completed task, and commits — one task, one commit.

When to use

Use /implement-next when you want to implement exactly one task from a plan file with full quality gates: TDD cycle, DA review loop, test suite pass, and a clean commit. It is the atomic unit invoked by /implement-all in a loop.

  • You have a plan file with one or more [ ] tasks and want to implement only the next one.
  • You want automated TDD enforcement: tests written first (red), then implementation (green), then full suite pass before committing.
  • You want a DA review loop to catch issues before the commit lands.
  • You are stepping through a plan manually, one task at a time, to inspect output between tasks.
  • You are debugging a failed /implement-all run and want to re-run a single task in isolation.

Relationship to /implement-all: /implement-all delegates to /implement-next for every uncompleted task in the plan. You can invoke /implement-next directly to implement one task at a time, or let /implement-all drive the loop automatically.

Relationship to scripts-plan: /implement-next calls plan-progress.sh in Step 1 to identify the next task and display progress, and calls check-task-commit.sh via /implement-all's verify step to confirm commit granularity.

Invocation

Pass the path to the plan file as the sole argument.

/implement-next [plan-file-path]

Parameters:

  • [plan-file-path] — (required) Relative path to the plan Markdown file, e.g. plans/FEAT-001-my-feature.md.

Steps executed for each task:

  1. Step 1 — Show progress: Runs plan-progress.sh to display current completion percentage and identify the next uncompleted task. If all tasks are complete (exit 1), stops and reports completion.
  2. Step 2 — Implement (TDD): Spawns the most appropriate agent to implement only this task. For code tasks: writes failing tests first (red), confirms they fail, implements enough code to pass (green), confirms all tests pass. For non-code tasks (docs, config): implements directly without a TDD cycle.
  3. Step 3 — DA review loop: Invokes /iterative-review with git diff HEAD and the task spec. DA agents review against acceptance criteria. All fixes stay as uncommitted working-tree changes — no DA commits.
  4. Step 4 — Run tests: Runs the full test suite. Spawns a fix agent for any failures; retries up to 3 times. Stops if the suite is not green after 3 attempts.
  5. Step 5 — Check off: Marks the implemented task and all completed sub-items [x] in the plan file.
  6. Step 6 — Commit: Stages implementation files and the updated plan file; commits with a message derived from the task content. --no-verify is forbidden.
  7. Step 7 — Report: Outputs a concise summary: what was implemented, test results, any deviations, and what was checked off.

Related tools

  • /implement-all — Drives the full plan loop by invoking /implement-next once per uncompleted task; use this when you want fully automated hands-off execution of all remaining tasks.
  • /iterative-review — The DA review loop invoked in Step 3 of every /implement-next run; spawns parallel DA agents and fix agents until no critical/major/moderate issues remain. (Forward reference — page not yet created; will resolve by Task 6.1.)
  • scripts-plan — The plan-progress.sh and check-task-commit.sh scripts used by /implement-next in Steps 1 and verify phases to track progress and enforce commit granularity. (Forward reference — page not yet created; will resolve by Task 6.1.)
