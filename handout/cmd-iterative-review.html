/iterative-review

← Back to index

Summary

Spawns multiple devil's advocate agents in parallel to review a target, then spawns fix agents to address all moderate-or-above issues, re-runs tests after each fix pass, and repeats until no critical, major, or moderate issues remain.

When to use

Use /iterative-review when you want review and automatic fixes applied in a loop — not just a flaw list. It keeps cycling until no critical, major, or moderate issues remain.

Key distinction from /da-review: /da-review identifies problems and stops — you fix them yourself. /iterative-review identifies and fixes all moderate-or-above issues automatically, re-running tests between each cycle. Choose /da-review when you want control over the fixes; choose /iterative-review when you want problems resolved automatically.

Typical trigger situations:

  • After implementing a task via /implement-next — to catch and fix issues before committing.
  • When a diff has multiple suspected issues and you want them all resolved in one shot.
  • When you need confidence that no critical, major, or moderate issues remain before merging.
  • As part of a multi-task plan execution loop (/implement-all calls this internally).

Invocation

Pass a target as the argument. If no argument is supplied, the command reviews the current uncommitted changes (git diff HEAD).

/iterative-review [target]

Parameters:

  • [target] — (optional) what to review: a file path, a plan file path, a git diff snippet, or a description. If empty, defaults to git diff HEAD.

Loop behavior: Each cycle spawns a minimum of 3 DA agents reviewing from different angles (correctness/edge cases, architecture/design, test coverage gaps). Issues are labelled with severity and a cycle-prefixed ID (e.g., C1-I-1). Fix agents address all Critical, Major, and Moderate findings. The loop stops when the consolidated issue list contains no Critical, Major, or Moderate items — or when the same root-cause issues reappear unchanged across cycles (unresolvable oscillation).

Severity rubric:

  • Critical — blocks correctness, security, or safety
  • Major — significant design flaw, missing requirement, or likely bug
  • Moderate — suboptimal but workable
  • Minor — style, naming, or nitpick (reported but not fixed automatically)

Note: Fix agents do NOT create git commits — all changes remain as uncommitted working-tree modifications.

Note on continuation: When invoked from /implement-next or /implement-next-cc, the calling skill includes its own "do not stop at the review verdict" reminder in its Step 3 — /iterative-review itself no longer prints a mandatory-continuation warning at the end of its summary.

Example 1 — Auto-fix issues in a feature diff (illustrative example)

Input:

/iterative-review $(git diff HEAD)

Output / Result (after 2 cycles):

## Cycle 1 — DA findings (3 agents)
Critical C1-I-1: `save_order()` called before payment confirmed — order can be
  saved without charge.
Major    C1-I-2: No rollback on partial failure between charge and order save.
Moderate C1-I-3: Retry count hardcoded to 3; should be env-configurable.

## Cycle 1 — Fixes applied
C1-I-1 fixed: moved `save_order()` call after payment confirmation check.
C1-I-2 fixed: added try/finally block with rollback on charge success + save failure.
C1-I-3 fixed: replaced magic number with MAX_RETRY env var (default 3).

Tests: 42 passed, 0 failed.

## Cycle 2 — DA findings (3 agents)
(no Critical, Major, or Moderate issues found)

## Review Summary
### Verdict
No critical, major or moderate issues remain.
### Remaining Open Issues
- Minor N1: variable `tmp` in line 47 could be renamed for clarity (not auto-fixed).

Example 2 — Review a plan file and fix gaps (illustrative example)

Input:

/iterative-review plans/FEAT-007-payments.md

Output / Result:

## Cycle 1 — DA findings
Major C1-I-1: Task 3.2 adds a `/refund` endpoint but no task covers updating
  ledger records on refund — creates an inconsistency in accounting reports.
Major C1-I-2: No rollback task defined if the DB migration in Task 4.1 fails.
Moderate C1-I-3: Acceptance criteria for Task 2.3 omit rate-limiting the
  charge endpoint.

## Cycle 1 — Fixes applied
C1-I-1 fixed: added Task 3.3 "Update ledger on refund" to the plan.
C1-I-2 fixed: added Task 4.2 "Rollback plan for DB migration failure".
C1-I-3 fixed: added rate-limit acceptance criterion to Task 2.3.

## Cycle 2 — DA findings
(no Critical, Major, or Moderate issues found)

## Review Summary
### Verdict
No critical, major or moderate issues remain.
### Changes Made
- C1-I-1 → Major → added Task 3.3 to plan
- C1-I-2 → Major → added Task 4.2 to plan
- C1-I-3 → Moderate → tightened Task 2.3 acceptance criteria

Related tools

  • /da-review — Single-pass review that finds issues without fixing them; use when you want to stay in control of the fixes yourself.
  • /implement-next (portable) — Implements one plan task end-to-end and calls /iterative-review internally as part of its Step 3 critical review.
  • /implement-next-cc (Claude Code) — Same caller pattern, hook-gated.
  • /implement-all / /implement-all-cc — Run /implement-next (or -cc) repeatedly until every plan task is done; each task's review loop uses /iterative-review.
  • /aaa — Evaluates quality against world-class standards with research and gap analysis; complements review by benchmarking against industry excellence, not just flagging flaws.
