scripts-plan

← Back to index

Summary

A group of shell scripts that track plan progress, enforce one-commit-per-task discipline, audit completed plan runs, and (in the Claude Code -cc variant) gate subagent turn-end on a commit via the SubagentStop hook — used directly by developers and automatically by /implement-next, /implement-all, /implement-next-cc, and /implement-all-cc.

When to use

Use these scripts whenever you are working with Markdown plan files that contain - [ ] / - [x] task checkboxes:

  • To see which task is next and the current completion percentage — use plan-progress.sh.
  • To count how many tasks remain before starting a run — use count-uncompleted-tasks.sh.
  • Inside /implement-next Step 7's self-verification — check-task-commit.sh verifies at least one non-merge commit was made.
  • At any later time, to audit a historical run independently — use audit-plan-run.sh.
  • The -cc variant scripts (implement-next-stop-gate.sh, implement-next-state-write.sh, implement-next-state-clear.sh) are invoked by /implement-all-cc and the SubagentStop hook — you don't call them by hand.

Invocation

User-invocable scripts

plan-progress.sh

Prints a formatted progress header for the next uncompleted task. Uses progress-header-phased.template (phased plans) or progress-header-flat.template (flat plans).

bash ~/.claude/scripts/plan-progress.sh <plan_file>

Exit codes: 0 = header printed + NEXT_TASK_* machine-readable lines; 1 = all tasks complete; 2 = usage error or file not found; 3 = no recognized task section heading.

count-uncompleted-tasks.sh

Counts top-level - [ ] lines within the recognized task section. Prints number_of_uncompleted_tasks=<N>.

bash ~/.claude/scripts/count-uncompleted-tasks.sh <plan_file>

Exit codes: 0 = success; 2 = usage error or file not found.

audit-plan-run.sh

Standalone post-run audit. Verifies the number of non-merge commits since <sha_start> matches the number of tasks completed in this run (current completed minus pre-run completed).

bash ~/.claude/scripts/audit-plan-run.sh <plan_file> <sha_start>

Exit codes: 0 = PASS; 1 = VIOLATION or mismatch; 2 = usage or git error.

verify-run-commits.sh (deprecated)

Verifies exactly <expected_count> non-merge commits were created since <sha_start>. Lighter than audit-plan-run.sh — does not diff the plan file. Deprecated — replaced by audit-plan-run.sh; still shipped for backwards compatibility.

bash ~/.claude/scripts/verify-run-commits.sh <sha_start> <expected_count>

Exit codes: 0 = PASS; 1 = VIOLATION; 2 = usage or git error.

check-task-commit.sh

Verifies at least one non-merge commit with non-empty file changes was created since <sha_before>. Auto-called by /implement-next Step 7 (portable) or by /implement-next-cc parent-level checks.

bash ~/.claude/scripts/check-task-commit.sh <sha_before>

Exit codes: 0 = PASS; 1 = VIOLATION; 2 = usage or git error.

Claude Code -cc variant scripts (hook + sentinel — not user-invoked)

implement-next-stop-gate.sh

SubagentStop hook script used by /implement-all-cc. Reads the sentinel file at $CWD/.claude/implement-next-state.json, filters on agentId (so nested sub-sub-agents pass through), and refuses to let the outer /implement-next-cc subagent end its turn until a new commit exists since START_SHA. Registered in ~/.claude/settings.json under hooks.SubagentStop.

implement-next-state-write.sh

Sentinel writer invoked by /implement-all-cc Step 4 (arm the gate) and /implement-all's portable parent (post-spawn), and by the standalone /implement-next-cc child after a Step 0 R-Fresh dispatch. Writes $CWD/.claude/implement-next-state.json atomically (tempfile + mv) using the v2 schema (schema_version: 2, plus branch_name, skill_variant, and review_abort_count) so the hook and the recovery classifier can read it.

# Default mode (parent-owned breadcrumb with real agentId):
bash ~/.claude/scripts/implement-next-state-write.sh <cwd> <sha_before> <plan_path> <task_name> <expected_agent_id> <skill_variant>

# --upsert mode (portable parent or standalone child; empty expected_agent_id permitted; read-merge preserves existing non-empty fields):
bash ~/.claude/scripts/implement-next-state-write.sh --upsert <cwd> <sha_before> <plan_path> <task_name> <expected_agent_id> <skill_variant>

# --increment-review-abort mode (child Step 3 second review abort; bumps review_abort_count durably):
bash ~/.claude/scripts/implement-next-state-write.sh --increment-review-abort <cwd>

The 6th positional arg skill_variant is one of "portable" or "cc" (default "cc" when omitted, for backward compatibility with pre-recovery callers).

implement-next-state-clear.sh

Sentinel cleaner. Called on success by the child's Step 7 self-verification (both variants), by the /implement-all-cc parent on Case B-after-rescue-still-fails / Case C / Case D / stuck-task / iter-count halt paths plus the final audit (defense-in-depth), and by /implement-all's portable parent at every halt path. Removes $CWD/.claude/implement-next-state.json so the next iteration's subagent is not gated prematurely.

bash ~/.claude/scripts/implement-next-state-clear.sh <cwd>

implement-next-triage.sh (recovery classifier)

Step 0 classifier called by both /implement-next and /implement-next-cc on every invocation. Reads the recovery breadcrumb plus repo state (HEAD, dirty/clean, branch, plan-file checked-task count) and dispatches to one of seven R-cases: R-Fresh (no prior interruption), R-A (uncommitted partial impl), R-B (committed but plan not checked off), R-AB (R-A and R-B combined), R-C (pre-impl TDD-red state — resume Step 2), R-Halt (corrupt breadcrumb in a dirty tree or plan deleted; exit 1), R-Stuck (review aborted twice; exit 1, manual recovery required). Prints KEY=VALUE lines (CASE, START_SHA, START_CHECKED, SHA_BEFORE, BRANCH_NAME, TASK_NAME, REVIEW_RANGE, STEP_2_RESUME, REVIEW_ABORT_COUNT) plus a human-readable RECOVERY: diagnostic line.

bash ~/.claude/scripts/implement-next-triage.sh <cwd> <plan_path> <current_skill_variant>

Exit codes: 0 = dispatched (R-Fresh / R-A / R-B / R-AB / R-C); 1 = halt (R-Halt / R-Stuck — corrupt breadcrumb in dirty tree, plan missing, review double-abort); 2 = usage error.

Bounded side effects (state hygiene): may delete $cwd/.claude/implement-next-state.json (on the committed-checked and auto-clear rows) and append to $cwd/.claude/recovery-anomalies.log (on the auto-clear row, with a one-time stderr notice suggesting the user add the log to .gitignore). Never spawns subagents, runs tests, or creates commits.

Example:

$ bash ~/.claude/scripts/implement-next-triage.sh "$(pwd)" "plans/FEAT-001.md" "cc"
CASE=R-Fresh
START_SHA=4f80ce924
START_CHECKED=14
TASK_NAME=**File**: `handout/scripts-plan.html`
RECOVERY: R-Fresh detected. sha_before=, head=4f80ce924, dirty=true. No prior breadcrumb; proceeding to Step 1.

Internal dependencies (called by user-invocable scripts — not invoked directly)

  • task_section.awk — AWK fragment that activates when the parser enters a recognized task section heading (## Tasks, ## Task, or ## Task breakdown). Called by plan-progress.sh, count-uncompleted-tasks.sh, and audit-plan-run.sh.
  • progress-header-phased.template — Mustache-style template for phased plans (plans with ### phase headings). Called by plan-progress.sh.
  • progress-header-flat.template — Mustache-style template for flat plans (no phase headings). Called by plan-progress.sh.

Example 1 — Check plan progress with plan-progress.sh

Input:

bash ~/.claude/scripts/plan-progress.sh plans/FEAT-001-skill-command-script-handouts.md

Output / Result:

Progress   : [█████████░░░] 79%  (15/19 tasks, 3/6 phases)
Next task  : 4.1  **File**: `handout/scripts-plan.html`

NEXT_TASK_NAME=**File**: `handout/scripts-plan.html`
NEXT_TASK_PHASE=4
NEXT_TASK_NUMBER=1

Exit code 0. The NEXT_TASK_* lines are machine-readable and parsed by /implement-next in Step 1 to identify the task to implement.

Example 2 — Audit a completed run with audit-plan-run.sh

Input:

bash ~/.claude/scripts/audit-plan-run.sh plans/FEAT-001-skill-command-script-handouts.md abc1234

Output / Result (PASS):

Plan file : plans/FEAT-001-skill-command-script-handouts.md
sha_start : abc1234
Tasks completed in this run (top-level) : 4
Non-merge commits since sha_start : 4

PASS: 4 commits for 4 completed tasks — counts match

Output / Result (VIOLATION):

Plan file : plans/FEAT-001-skill-command-script-handouts.md
sha_start : abc1234
Tasks completed in this run (top-level) : 4
Non-merge commits since sha_start : 3

VIOLATION: 3 commits for 4 completed tasks — counts do not match

A VIOLATION means at least two tasks were bundled into a single commit, breaking the one-task-one-commit rule. Exit code 1.

Related tools

  • /implement-next (portable) — Calls plan-progress.sh in Step 1 and check-task-commit.sh in Step 7's self-verification.
  • /implement-next-cc (Claude Code) — Same caller pattern; additionally gated by the SubagentStop hook (implement-next-stop-gate.sh) when invoked from /implement-all-cc.
  • /implement-all (portable) — Calls plan-progress.sh per iteration and audit-plan-run.sh as a final audit.
  • /implement-all-cc (Claude Code) — Same callers plus the sentinel pair implement-next-state-write.sh / implement-next-state-clear.sh around each subagent spawn.
