/wrap-up

← Back to index

Summary

Find out whether the session is actually done — tests run, commit audited, tracker checked — before you close the tab. Verifies freely; never commits, fixes, or updates anything without an explicit yes to that exact action.

When to use

You've been coding for hours and something nags: did we finish? Did the tests pass? Did we commit? Use /wrap-up at the end of any work session to get an honest answer — "done and clean", "blocked by a defect", or "clean but N steps left" — before you stop.

Trigger phrases (from SKILL.md):

  • "wrap up"
  • "are we done"
  • "what's left"
  • "did we commit everything"
  • "close out"

The non-negotiable rule: tests and devil's-advocate run every time — "it was a small change" and "I'm confident it's fine" are not valid skips. The only acceptable non-run is a hard block (e.g. Docker not up), which is reported as ❓, never silently omitted.

Invocation

Say the trigger phrase or invoke the skill directly:

/wrap-up

No parameters. The skill reads session state, git status, and the relevant tracker item itself.

What it runs automatically (no asking): test suite + /da-review on this session's diff.

What it asks before doing: committing, fixing code, updating the tracker — any write. One yes per action.

Input:

wrap up

Related tools

  • /da-review — The devil's-advocate pass /wrap-up runs internally; use it standalone for a one-off adversarial review mid-session.
  • /iterative-review — Heavier review loop with auto-fix passes; reach for it when /wrap-up's DA finds more than one or two issues.
  • /implement-next — The tool that precedes wrap-up: builds the next plan task test-first and commits it.
